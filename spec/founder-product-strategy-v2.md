## 1. 전략 한 줄

Dochi는 `한 앱` 안에서 `가족 홈 어시스턴트`와 `개인 어시스턴트`를 역할 프로필로 운영하며, 허락 기반 적극성으로 반복 가치를 만들고 비용을 통제한다.

---

## 2. 창업자 관점 핵심 판단

1. 지금 중요한 건 앱 분리가 아니라 `활성화(Activation)`다.
2. 분리는 복잡도(동기화/권한/운영)를 늘리고, 초기 PMF 속도를 늦춘다.
3. 따라서 현재 전략은 `Single App, Multi-Profile`.
4. 분리 판단은 기술이 아니라 지표 트리거로 한다.

---

## 3. Single App vs Split App 의사결정

## 현재 결정
- **채택**: Single App (호스트/어시스턴트 역할 프로필)
- **보류**: Host App + Assistant App 분리

## 왜 지금은 단일 앱인가
1. 이미 서비스 초기화/실행 경로가 하나의 런타임으로 결합되어 있다.
2. 현재 병목은 구조가 아니라 온보딩, 설정 완성도, 프로액티브 품질이다.
3. 분리 전에 단일 앱에서 가치 루프를 먼저 증명해야 한다.

## 분리 Go/No-Go 트리거
아래 3개 중 2개 이상 충족 시 분리 검토:
1. 백그라운드 상시 실행 요구가 core value의 50% 이상
2. 가족 안전 정책상 권한 격리가 법적/운영상 필수
3. 호스트와 어시스턴트의 릴리즈/과금/세일즈 채널이 실질적으로 분리

---

## 4. 제품 포지셔닝

## Primary ICP
1. 가족 중심 사용자 (부모/가정 운영)
2. 생산성 중심 개인 사용자

## 제품 약속
1. 필요한 순간 먼저 찾아오되, 실행은 허락을 구한다.
2. 가족과 개인 맥락이 섞이지 않게 기억과 행동을 분리한다.
3. 토큰/비용을 사용자가 이해 가능한 수준으로 투명하게 통제한다.

---

## 5. 운영 프로필 체계 (양극단 회피)

## 5.1 프로필 정의
1. `조용한 동행형`
2. `가족 홈 어시스턴트형` (기본 권장)
3. `개인 생산성형`
4. `자동형-안전범위`

## 5.2 가족 홈 어시스턴트형 기본 정책
1. 제안은 적극적, 실행은 허락 기반
2. 리마인더/칸반 등록 제안 우선
3. 보호자 확인 필요한 작업 분리
4. 아동 프로필은 빈도/범위 보수 적용
5. 야간에는 요약형 알림만 허용

---

## 6. 가치 루프 설계

## 6.1 Activation Loop
1. 온보딩 중 Quick Seed (리마인더 1개, 칸반 카드 1개, 자동화 1개)
2. 첫 제안 카드 노출
3. 첫 수락 경험 완료

## 6.2 Daily Value Loop
1. Heartbeat가 기회(TaskOpportunity) 생성
2. 제안 카드에서 허락 획득
3. 리마인더/칸반/자동화로 실행
4. 완료 결과를 메모리/관심사에 반영

## 6.3 Retention Loop
1. 제안 수락/연기/거절 학습
2. 다음 제안 품질 개선
3. 사용자 체감 “쓸모 있음” 강화

---

## 7. 제품 원칙 (Exit 관점)

1. Activation 먼저, 기능 추가는 나중
2. 허락 없는 고비용 실행 금지
3. 설정 미흡은 사용자가 아니라 앱이 발견하고 복구
4. 신뢰를 잃는 과한 자동화보다, 느리더라도 높은 예측 가능성
5. 모호한 기능보다 측정 가능한 루프 우선

---

## 8. 설정 완성도 전략 (Setup Health System)

## 8.1 Setup Health Score
- 0~100 점수로 상태 표시
- `required`, `recommended`, `optional` 분리

## 8.2 자동 복구 동선
1. 부족 설정을 상단 배너로 제시
2. 원클릭 이동
3. 가능한 항목은 원클릭 자동 해결

## 8.3 최소 required 예시
1. API 키 또는 로컬 모델 준비
2. 프로필(가족형/개인형) 선택
3. 프로액티브 기본 정책 설정
4. 자동화 1개 이상

---

## 9. 신뢰/안전 운영 모델

1. 제안 단계: 항상 근거와 예상 영향 표시
2. 실행 단계: 민감 동작은 보호자/사용자 확인
3. 감사 단계: 최근 자동 제안/실행 로그 확인 가능
4. 차단 단계: 유형별 끄기/쿨다운/야간 제한

---

## 10. 비용/리소스 운영 모델

1. Stage A: 0-token 룰 평가
2. Stage B: 저비용 요약/문구
3. Stage C: 고비용 실행 (허락 후)
4. 예산 임계값(50/80/100%) 상태별 자동 감쇠
5. 프로액티브 일일 캡과 중복 제안 방지 기본 적용

---

## 11. 기존 구현 미흡 마무리 (필수 선행)

1. 프로액티브 일일 캡 실적용
2. activity 신호 확장 (메시지 전송 외 상호작용 포함)
3. 프로액티브 알림 정책 이중화 해소
4. Heartbeat 설정 내 프로액티브 토글 UX 결함 수정
5. 중복 설정 화면 책임 재정의

관련 상세는 [onboarding-proactive-activation-plan.md](./onboarding-proactive-activation-plan.md) 참조.

---

## 12. 12주 실행 로드맵 (Founder Sequencing)

## Phase 1 (주 1~4): Activation 기반 만들기
1. Quick Seed 온보딩
2. Setup Health Score
3. 일일 캡/정책 통합

## Phase 2 (주 5~8): 가족형 가치 강화
1. 가족 홈 어시스턴트 루틴 템플릿
2. Heartbeat -> TaskOpportunity 엔진
3. 허락형 등록 플로우 최적화

## Phase 3 (주 9~12): 품질과 성장
1. 제안 랭킹 고도화
2. 프로필별 A/B 실험
3. 리텐션/비용 최적화

---

## 13. KPI 트리 (투자/엑싯 관점)

## North Star
- `WAO (Weekly Assisted Outcomes)`: 주간에 유저가 수락/실행한 유의미한 도움 건수

## Input KPI
1. Install -> First Value 전환율
2. Time to First Value
3. Quick Seed 완료율
4. Setup Health 80점 이상 도달률

## Quality KPI
1. Proactive Acceptance Rate
2. 불필요 제안 피드백률
3. 가족형 프로필 보호자/아동 만족도 편차

## Efficiency KPI
1. Proactive Token Spend Ratio
2. 예산 초과 사용자 비율
3. 실행 성공률/실패률

## Retention KPI
1. D1 / D7 / D30
2. 가족형 프로필 주간 재방문율
3. 자동화 1개 이상 유지율

---

## 14. 수익화 가설 (현 단계 초안)

1. Free: 기본 대화 + 제한된 프로액티브
2. Plus: 가족 프로필/자동화/고급 제안
3. Pro: 멀티 디바이스/고급 통합/MCP 확장

가격 이전에 우선 검증할 것:
1. 주간 반복 가치
2. 비용 대비 효용
3. 가족형 시나리오의 장기 유지성

---

## 15. 엑싯 관점에서 보는 모트(Moat)

1. 가족+개인 맥락을 안전하게 다루는 운영 정책 레이어
2. 허락 기반 적극성 데이터셋(수락/거절/피로도)
3. 리마인더/칸반/메모리/자동화의 결합된 실행 그래프
