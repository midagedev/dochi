# 02. Architecture Principles

## 1) CONCEPT 정렬

Dochi의 원래 컨셉을 리라이트의 하드 제약으로 둔다.

1. 컨텍스트가 핵심
2. 선언적 에이전트
3. 로컬 실행
4. 안전한 권한 분리

이 네 가지를 훼손하는 구현은 기술적으로 쉬워도 채택하지 않는다.

## 2) 리라이트 핵심 원칙

### A. 엔진 표준화

- 에이전트 추론/도구 루프는 Claude Agent SDK로 일원화한다.
- 앱 코드에서 tool loop를 재구현하지 않는다.

### B. 도메인 독립성

- Dochi의 경쟁력은 "기억/맥락/디바이스 오케스트레이션"이다.
- SDK 교체 가능성을 위해 도메인 모델과 런타임 모델을 분리한다.

### C. 로컬 우선 실행

- 툴 실행과 민감 맥락 조합은 가능한 한 디바이스에서 수행한다.
- 클라우드는 동기화/라우팅/협업 상태 공유에 집중한다.

### D. 단일 권한 진실원

- 권한 판단은 한 경로에서만 최종 판정한다.
- 중복 정책 엔진(앱 레이어/런타임 레이어 이중 판단)을 피한다.

### E. 관측 가능성 내장

- 각 툴 호출, 훅 결과, 세션 상태 전이를 이벤트로 남긴다.
- "왜 이 결정을 했는지"를 운영 로그에서 재구성 가능해야 한다.

## 3) 비기능 요구사항

### 성능

- 음성 명령 첫 응답(첫 partial) 목표: 2.0초 이내
- 일반 도구 호출 왕복 목표: 3.0초 이내 (로컬 도구 기준)

### 안정성

- 런타임 sidecar 비정상 종료 시 자동 재기동
- 세션 복구 가능률 99% 이상 목표

### 보안

- Restricted 도구는 사용자 명시 승인 없이는 실행 금지
- 메모리 동기화 시 개인 컨텍스트는 workspace 공유 금지

### 운영성

- 세션/툴/권한 이벤트를 구조화 로그(JSON)로 저장
- 장애 재현을 위한 최소 24시간 로컬 진단 로그 보관

## 4) 안티 패턴

- 기존 loop + SDK loop를 동시에 운용하는 이중 엔진
- 채널별(음성/메신저/UI) 별도 권한 규칙
- 에이전트 설정을 코드 하드코딩으로 분산
- 동기화 계층에서 도구 실행까지 담당하게 만드는 설계

## 5) 아키텍처 결정 규칙

새 설계 제안은 아래 질문을 통과해야 한다.

1. 이 결정이 컨텍스트 품질을 높이는가?
2. 권한 경계를 더 명확히 만드는가?
3. SDK 표준 기능으로 대체 가능한데 재구현하고 있지 않은가?
4. 멀티 디바이스에서 동일 동작을 재현 가능한가?
5. 장애 시 사람이 이해 가능한 로그를 남기는가?
